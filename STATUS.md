# Cuaderno Status

Snapshot of development progress as of the most recent merge. For per-PR detail see [`CHANGELOG.md`](CHANGELOG.md); for the underlying plan see [`docs/implementation-plan.md`](docs/implementation-plan.md).


**Current release**: 0.28.0 (2026-07-13). v0.28.0 makes AI agents aware of two things they were blind to. A new read-only **`list_note_types`** MCP tool reports every note type — the 12 built-ins plus any config-defined `[note_types.*]` custom type — with its folder, required/optional fields, typed `[schemas.*]` field specs (name, type, default, allowed values), template, and supplied placeholders, so an agent can learn a vault's custom types before `create_custom_note` instead of reading `config.toml` or guessing (the create tool's schema deliberately can't enumerate them); it's assembled by a new `TypeRegistry::describe_all` and served on the read-only surface. And a new **`cuaderno://note/<path>` deep link** opens that note in the desktop reader — so an agent can hand the user a clickable "open in the reader" link built from any note path a tool returns; the path is confined by the reader's `VaultPath` guard (a `..`/absolute link can't read outside the vault), cold-start launches are buffered and drained on mount so the note opens even when the link launches the app, and daily/weekly/evidence/tracking are correctly reported as append-only (#419). v0.27.0 makes the desktop app feel native to macOS and fully themeable. The main window takes on native chrome — an overlay title bar with inset traffic lights, an `NSVisualEffectView` vibrancy sidebar, and draggable title strips (the main window gains `core:window:allow-start-dragging`) — over a system-font, cool-neutral palette. A reworked **Settings** panel (`⌘,`) gains an **Appearance** section (theme mode system/light/dark, a curated palette of default/warm/cool/high-contrast, an accent colour, sidebar width, and a reduce-transparency toggle) and a **Reading** section that tunes the markdown reader's text size, column width, body font, and line spacing — headings scale in `em`, so KaTeX maths tracks the text size for free. Every choice is stamped as a `data-*` attribute mapped to a CSS variable, persists per-device in `localStorage`, and previews live; the seven context hues are never touched by the accent axis. For anything the presets don't cover, a hand-editable `.cuaderno/custom.css` — seeded with a documented template — is injected after the bundled styles (reapplied on window focus) and can redefine any design token: accents, fonts, heading sizes, or what "narrow" means, editable in-app via a CodeMirror CSS editor (applies on save) or in an external editor, backed by new `read`/`init`/`write`/`open_custom_css` Tauri commands (#417). v0.26.0 turns the desktop note reader into a comfortable reading and editing surface for maths-heavy research notes. The right-anchored slide-in reader is replaced by a centred full-page reader at `/note/<path>` — a ~72ch reading column in the content area, the sidebar in place, history back/forward between notes, and a Read/Edit toggle over the CodeMirror-6 posture-B editor first spiked in #413 — reached by every surface that used to summon the drawer. LaTeX maths renders via KaTeX (`$…$`/`$$…$$`, stylesheet and fonts vendored offline so nothing hits a CDN under the app CSP), and note-embedded images render inline — a `read_note_asset` command resolves an image src relative to its note, confines it with `VaultPath`, and serves the bytes as a `data:` URI. The reader's actions sit in a sticky header, the frontmatter "Properties" block is collapsible with an aligned key/value grid, and the inbox drawer and calendar day panel gain an in-app "open" alongside "Open in editor" (#413, #415). The Weekly review's Wins step now shows the week's completed actions as celebration cards, and every log-card surface gains a remembered Oldest/Newest-first toggle (#412). And two vault-graph/card fixes: a bare folder wikilink `[[portfolios/<slug>]]` now resolves to the folder's `_index.md`, so every portfolio link in a log clicks (previously dead), and an overflowing Home project-card next-action caps at a fixed height with a more/less toggle (#414). v0.25.1 fixes the v0.25.0 mouse back/forward feature on macOS: the side buttons did nothing because this Mac's mouse fires them as horizontal **swipe gestures** (`NSEventTypeSwipe`), not `otherMouseUp` buttonNumber 3/4 — root-caused live with an instrumented dev build. The native `NSEvent` monitor now maps a swipe's `deltaX` direction to back/forward (verified against the mouse) while keeping the buttonNumber path, and trackpad page-swipes navigate too (#410). v0.25.0 adds two desktop conveniences. **Back/forward navigation** — `⌘[`/`⌘]` (Ctrl elsewhere) and the mouse's back/forward side buttons step through view history like a browser; since macOS WKWebView doesn't deliver those mouse buttons to the web layer (tauri#10936), a native `NSEvent` monitor bridges them via `nav://back`/`nav://forward` events, with the keyboard path working on every platform (#407). And a **Settings panel** on `⌘,` (the macOS Preferences convention): a calm modal consolidating the app-level preferences that had been blind footer toggles — theme as a segmented control showing the current choice, a labelled metrics switch, and a jump to the vault config editor (#408). v0.24.0 continues the desktop-UI presentation work. The shell **note reader** (the slide-in opened from a wikilink or backlink) now shares the calendar's sectioned rendering via a shared `SectionedBody`, so a note reads identically wherever it opens — titled `##` sections and the `## Logs` history as timestamped cards (the reader omits the calendar's fixed-height Logs cap, since its own panel is the single scroll) (#404). Under the hood, two open-coded idioms were consolidated into shared primitives — the `shortDate` formatter (seven call sites → `lib/dates`) and the uppercase section-label heading (~23 sites → `components/ui/SectionHeading`) — a byte-identical refactor with no behaviour change (#405). v0.23.0 is a desktop-UI presentation pass. A note's frontmatter in the reader moves out of a bare chip row (which read like the note's first heading) into a distinct, labelled `MetaPanel` "Properties" block, and a new reusable `LogCard` gives timestamped log lines a "when + what" hierarchy — replacing the flat bullet list under a project's *Recently in your logs* (#401). The desktop **calendar** now presents a day's note in sections rather than as a raw markdown blob: it parses the note (frontend-side; the domain still hands back one blob) into the separated frontmatter strip, each `##` section under a clear title, and the append-only `## Logs` history as a scrollable stack of timestamped log cards (wikilinks preserved) — and demotes the month grid from a fixed column to a hideable **Pick a date** picker so the note leads. Weekly and monthly notes get the same sectioned treatment (#402). Pure view changes — no domain or wire-format change. v0.22.0 bundles three vault-graph refinements: **note slugs are now globally unique** (#225) — creating a project, action, commitment, question, or custom note whose slug is already taken anywhere in the vault suffixes it `-2`/`-3`/… instead of erroring, so a note that later relocates (an action archived to `_done/`, a project parked) keeps its `[[type/slug]]` backlinks rather than degrading on a stem collision (stewardships excluded — they don't relocate and their flat-vs-expanded guard is worth keeping; forward-only, pre-existing same-stem notes keep their slugs); **backlinks now catch frontmatter wikilinks** (#395) — the index scans a note's frontmatter string values for `[[wikilinks]]` alongside its body, so `find_backlinks` (and thus `project_backlinks`/`question_backlinks`) now includes a project's `core_question:`, a portfolio's `project:`, and an evidence note's `origin:`, which in particular populates the Strategic questions grid's project chips for the common `core_question:` case; and the **backlink chips on the desktop Strategic questions grid** themselves (#354): each question card surfaces the notes that reference it — projects that wikilink the question (routed to the project detail page), portfolios (routed, both slug-shared and those linked via `link_portfolio_to_question`), and evidence (opening in the reader) — via a new `question_backlinks` domain method mirroring `project_backlinks` (now populated for `core_question:` and other frontmatter links via #395, above). v0.20.1 finished the config-reload UX (#384): when a busy vault defers an otherwise-valid external `config.toml` edit, the desktop app now shows a distinct, calm "the vault was busy — it'll be re-applied on the next change" banner instead of silently keeping the last good config, and a successful in-app save (or any reload) clears a lingering invalid/deferred banner. v0.20.0 was a token-cap-safety pass over the MCP context tools with one breaking payload change: `get_weekly_context` state changes now carry only the `new_state` gist (`old_state` dropped — it is auto-logged to the daily note before every overwrite, so a review reconstructs it from the log) (#351), and `get_project_context` bounds its growable fields — `recent_mentions` to the 50 most-recent, `body_markdown` to a generous 20k-char safety valve, and each `backlinks` group to 100 — each drop observable and the full data one `read_note`/`read_daily_note` away, while the CLI and desktop app keep the full uncapped data (#352, #388). v0.19.1 fixed a desktop config-reload papercut (#372): a transiently held vault write lock during an external `config.toml` reload no longer surfaces a false "config.toml has an error" banner — the watcher now tells a genuinely invalid config from transient contention, retrying the reload once and otherwise keeping the last good config silently until the next config edit. v0.19.0 **completes the desktop Config editor** (#365, PRs 4-5): the Config view gains a **Raw / Form** toggle. Raw is the raw `config.toml` editor; Form is a structured editor that adds, edits, and removes custom note types and their `[schemas.<type>]` field declarations via a surgical `toml_edit` writer that preserves comments, key order, and the `[variables]` block byte-for-byte, flowing through the same validate-first -> compare-and-swap -> write -> live-reload gate as Raw (so a form edit can no more brick the vault than a raw one). External `config.toml` hand-edits now also apply **live**: the app rebuilds its vault on the fly and — on an invalid edit — keeps the last good config and shows a calm, non-red banner. It also fixes a staleness off-by-one (#379): stuck-project "state untouched for N days" is now counted in local days, not UTC, so a project touched today is no longer flagged one day stale in the hours after local midnight. v0.18.0 added **editing `config.toml` from the desktop app** (#365, PRs 1-3): a new **Config** view (Browse group) that shows the raw config, type-checks it against the same validation `Vault::new` runs, and lets you edit and save it in-app. The save path validates FIRST as a hard precondition (an invalid config is never written, so the vault can never be bricked), compare-and-swaps on a content hash to catch a concurrent hand-edit, writes verbatim (comments preserved), and applies the change LIVE — the managed vault is now hot-swappable (`ArcSwap`), rebuilt on the same store/index without a restart. v0.17.0 completes **config-driven frontmatter** (#301 Phase 2): a generic `set_frontmatter` setter — CLI `cdno frontmatter set <note> <key> <value>` and the `set_frontmatter` MCP tool (45th) — writes a typed, `settable`-declared field and its `.cuaderno/index.db` row in one atomic transaction (no hand-edit, no index desync), rejecting unknown/non-settable/reserved (`type`/`status`/period-key) fields and type-checking the value; a `log_on_change` field also stamps a `key: old -> new` line into today's daily. So `cdno frontmatter set today meds true` just works. v0.16.0 lands **config-driven frontmatter, Phase 1** (#301): a built-in note type can declare typed extra fields under `[schemas.<type>.fields]` (`type` = bool/int/string/date, optional `default`/`values`), and those fields now (a) populate their `default` into a note's frontmatter at creation — when a custom `.cuaderno/templates/<type>.md` references them — with engine and `[variables]`/prompt values taking precedence over the default, (b) are recognised by the desktop Templates editor (no more "renders literally" on a declared field), and (c) are type-checked by `cdno lint`. The `set_frontmatter` setter (toggling a field without a hand-edit) is the planned Phase 2. v0.15.0 adds a desktop **Templates view** (#357) at `/templates`: browse every note type's template (built-in default vs custom override vs a config-defined type with none yet), edit it in-app and save to `.cuaderno/templates/` (the app's first free-text file editor; editing a built-in-backed type transparently writes the override), or create a starter for a custom type. A placeholder-reference panel lists each type's available variables — including a config-defined custom type's own schema fields — and the editor flags any unknown `{{placeholder}}` with a calm, non-blocking warning. v0.14.0 adds startup observability: `cdno-mcp-server` (and the stdio server) now log the resolved local timezone offset and a sample local timestamp at startup, so a silent UTC fallback — a container without `tzdata`/`TZ`, which lands daily-log timestamps in the wrong zone — shows up immediately in the logs instead of days later (#309). v0.13.0 fixes two daily-use papercuts: a customised daily template referencing `{{day_name}}` or `{{week}}` now renders them (the scaffold gained both, sharing the weekly note's ISO-week formatting) instead of leaving the placeholders literal (#300), and `get_weekly_context` is bounded so it no longer overflows the MCP client's token cap — state changes carry a ~200-char gist rather than full before/after bodies, and logs cap at the 100 most-recent lines (#298). v0.12.0 polishes the desktop app: the Strategic questions grid shows **portfolio link chips** per question (#339), Stewardship Detail draws integer count/volume series as calm **column charts** while continuous measures keep the line (#337), and an ambiguous completion selector now opens a **disambiguation picker** to choose the intended candidate instead of ## What works today

Reachable from the terminal via `cdno`:

- `init` — scaffold a vault
- `log` / `lint` / `capture` — daily-log writes, validation, inbox capture
- `project create / state / park / activate / list / show / milestone {add,done} / waiting {add,resolve}`
- `action add / promote / complete / list` (bullet form + manifest note form)
- `commit create / complete` and `commitments` aggregated view
- `orient` / `status` — morning views
- `portfolio create / list / show` and `file` (file evidence into a portfolio; `--attach <path>` files a non-markdown artefact — PDF/image/video — by copying it in and scaffolding a linked evidence stub, `--move` to relocate instead, #154)
- `question create / park / answer / retire / activate` and `questions` (active grouped by domain)
- `stewardship create / list / show / add-periodic` and `track <activity>`
- `search <query>` — full-text content search across the vault, ranked best-first, with optional `--type` / `--from` / `--to` / `--portfolio` / `--limit` filters
- `completions <shell>` — emit a shell-completion shim (bash, zsh, fish, elvish, powershell) with **dynamic vault-aware tab completion** for slug-valued flags (`--project`, `--portfolio`, `--stewardship`, `--slug` on project/question verbs)

Reachable from Claude via MCP (`cdno-mcp` binary):
