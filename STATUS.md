# Cuaderno Status

Snapshot of development progress as of the most recent merge. For per-PR detail see [`CHANGELOG.md`](CHANGELOG.md); for the underlying plan see [`docs/implementation-plan.md`](docs/implementation-plan.md).


**Current release**: 0.32.1 (2026-07-24). v0.32.1 is a lint fix (#452): a wikilink or embed pointing at an attachment — a pasted image, a filed PDF — is no longer reported as a broken link. Attachments are deliberately never indexed, so `cdno lint` resolved every such target against the index and failed; one note with 54 pasted screenshots produced 54 warnings for files all present on disk. A target that names an existing file under the vault is now resolved against the filesystem — relative to the linking note first (so it keeps working when the note sits inside a grouping subfolder, not only at the portfolio root), then relative to the vault root — before a link is called broken, using `read_bytes` as the file probe so a directory is never mistaken for a file and `..` targets stay confined to the vault. The fallback is scoped to targets that look like a file (an embed, or one carrying an extension) so a genuinely broken plain `[[note]]` link is never masked, and a missing `![[embed]]` reads as a missing file rather than a link that "resolves to no note". v0.32.0 is the desktop-UI epic (#439): the app now reads as the Research Logbook Method rather than a flat pile of screens. The sidebar becomes three tracks bound by a cadence -- **Rhythm** (Today, Calendar, Weekly, Monthly), **Operations** (Projects, Actions, Commitments, Stewardships), **Inquiry** (Questions, Portfolios) -- with the 5-slot project cap shown inline; **Strategic** is renamed **Monthly** (with a `/strategic` redirect) and Templates + Config move into Settings behind `Cmd+,` guarded against discarding unsaved drafts (#444). **Questions** get a surface of their own, grouped by domain, status changeable in place (#443). **Today** leads with the daily note and a "Now" band naming the action you are in the middle of, read back from the log so a CLI or MCP start counts too (#442); the **project map** leads with its Current State and caps its backlinks/log-mentions (#441). **Actions** becomes a working surface -- a project rail with counts, sticky context/energy filters including the previously-dropped "untagged" bucket, one-line rows (#445). The **Calendar** keeps the month pinned beside the note and marks today wherever you page (#446). **Commitments** answer "when" -- This week / Next week banding, a month-grid view, a settable horizon replacing the hardcoded 90 days (#447). **Stewardships** show status not progress -- quietest-first, a gone-quiet filter, the shared staleness ink ladder, logging from the row (#448). The **weekly review** shows its shape (a labelled stepper) and hands back the wins as cards you tick, edit, reorder and remove with a lossless markdown round-trip (#449). And the **monthly review** becomes a stepped ritual that writes wins, themes and next-month focus to the monthly note via a new `save_monthly_section` command -- the artefact a cadence needs (#450). Portfolio indexing was fixed underneath it all: markdown filed into a portfolio is treated as an attachment, not an unindexable note (#440, #451). v0.31.0 puts a **configurable length ceiling on a project's Current State** so agent-driven updates stop sprawling that snapshot into a running narrative. Two new `[vault]` config keys cap the `## Current State` section: `max_state_chars` (Unicode scalar count, default `500`, `0` disables) and `state_overflow` (`reject` the default, `warn`, or `off`). It is enforced once in `update_project_state`, so the CLI, MCP, and desktop share one rule: `reject` blocks the write with an actionable message, `warn` writes but returns a non-fatal advisory the surfaces show (CLI stderr, the MCP result message, a desktop toast) via a new `WriteOutcome.warnings` channel. Existing over-limit states are grandfathered and can be trimmed incrementally — any edit that does not grow an already-over state is accepted even while still over, so you are never forced under the cap in a single edit; only a *new* overflow, or growing an existing one, is blocked. The previous state is still auto-logged to the daily on every update, so the long-form history is never lost (#436). v0.30.0 adds an easy way to **add a log entry to today by hand** from the desktop app: a one-line composer sits on the **Today** landing (always today) and at the top of the **calendar's daily panel** (shown only for today), appending to today's `## Logs` via the same append-only `log_quick` verb the global capture window and CLI write to — in the calendar the new entry appears immediately as a log card. The composer is guarded against a fast/held-Enter double-submit, and `get_today` is now invalidated on the day-change tick so a surface left open across midnight re-reads the new day (#433). v0.29.2 brings the app's rich markdown surfaces to two more places (desktop UI): the Weekly review's **Wins** step now composes in the CodeMirror markdown editor (syntax highlighting, line wrapping, the shared theme) instead of a plain textarea, and **Inbox** captures render as formatted markdown — headings, lists, single-newline line breaks, and clickable wikilinks that open their target — rather than an inert wall of raw text, clipped to a preview height so a long capture doesn't dominate the narrow drawer (#431). v0.29.1 makes a note's **external links work** in the desktop app: `http(s)`/`mailto` links rendered as inert, muted dead text (the webview can't navigate out and no opener was wired) and now render as real links that open in the default browser (or mail client) on click — left-, cmd-, and middle-click are intercepted at the web layer and handed to a scheme-validated backend opener (`http`/`https`/`mailto` only, so a note can't launch a `file://`/`javascript:`/custom-scheme URL); wikilinks and relative links are unaffected (#427). v0.29.0 finishes the desktop **Config editor** and polishes note reading. The Config form's structured view now edits the two pieces that still needed the Raw editor. A schema field gains **Settable** (whether `set_frontmatter` may change it) and **Log changes to daily** toggles, persisted through the surgical `toml_edit` writer that had been silently dropping the reserved keys — so a form toggle now actually sticks across a re-parse, with `Log changes to daily` gated on (and cleared with) `Settable` (#422). And a new **Template variables** section edits the `[variables]` block: **Static** variables (available in every template) and **Prompted** variables (`[variables.prompt]`, asked for interactively), each a name→value row with add, inline edit, and remove — the serde-flattened block lifted into sorted named lists (ts-rs can't follow `flatten`) and written comment-preserving, with `prompt` refused as a static name since it is the prompt sub-table (#424). The desktop **calendar** gains a **Today** button next to *Pick a date* that jumps straight back to today's daily note (disabled while already on it), and note rendering now shows a single newline as a line break — matching Obsidian's default — so a standup's *Yesterday* / *Today* / *Due soon* lines stay on their own lines instead of collapsing into one paragraph, everywhere notes render (#425). Under the hood, the file watcher no longer reconciles the index twice on an external `config.toml` edit: a successful live rebuild's own reconcile is the single pass, with the standalone reconcile kept only on the failure paths where the old vault stays live (#421). v0.28.0 makes AI agents aware of two things they were blind to. A new read-only **`list_note_types`** MCP tool reports every note type — the 12 built-ins plus any config-defined `[note_types.*]` custom type — with its folder, required/optional fields, typed `[schemas.*]` field specs (name, type, default, allowed values), template, and supplied placeholders, so an agent can learn a vault's custom types before `create_custom_note` instead of reading `config.toml` or guessing (the create tool's schema deliberately can't enumerate them); it's assembled by a new `TypeRegistry::describe_all` and served on the read-only surface. And a new **`cuaderno://note/<path>` deep link** opens that note in the desktop reader — so an agent can hand the user a clickable "open in the reader" link built from any note path a tool returns; the path is confined by the reader's `VaultPath` guard (a `..`/absolute link can't read outside the vault), cold-start launches are buffered and drained on mount so the note opens even when the link launches the app, and daily/weekly/evidence/tracking are correctly reported as append-only (#419). v0.27.0 makes the desktop app feel native to macOS and fully themeable. The main window takes on native chrome — an overlay title bar with inset traffic lights, an `NSVisualEffectView` vibrancy sidebar, and draggable title strips (the main window gains `core:window:allow-start-dragging`) — over a system-font, cool-neutral palette. A reworked **Settings** panel (`⌘,`) gains an **Appearance** section (theme mode system/light/dark, a curated palette of default/warm/cool/high-contrast, an accent colour, sidebar width, and a reduce-transparency toggle) and a **Reading** section that tunes the markdown reader's text size, column width, body font, and line spacing — headings scale in `em`, so KaTeX maths tracks the text size for free. Every choice is stamped as a `data-*` attribute mapped to a CSS variable, persists per-device in `localStorage`, and previews live; the seven context hues are never touched by the accent axis. For anything the presets don't cover, a hand-editable `.cuaderno/custom.css` — seeded with a documented template — is injected after the bundled styles (reapplied on window focus) and can redefine any design token: accents, fonts, heading sizes, or what "narrow" means, editable in-app via a CodeMirror CSS editor (applies on save) or in an external editor, backed by new `read`/`init`/`write`/`open_custom_css` Tauri commands (#417). v0.26.0 turns the desktop note reader into a comfortable reading and editing surface for maths-heavy research notes. The right-anchored slide-in reader is replaced by a centred full-page reader at `/note/<path>` — a ~72ch reading column in the content area, the sidebar in place, history back/forward between notes, and a Read/Edit toggle over the CodeMirror-6 posture-B editor first spiked in #413 — reached by every surface that used to summon the drawer. LaTeX maths renders via KaTeX (`$…$`/`$$…$$`, stylesheet and fonts vendored offline so nothing hits a CDN under the app CSP), and note-embedded images render inline — a `read_note_asset` command resolves an image src relative to its note, confines it with `VaultPath`, and serves the bytes as a `data:` URI. The reader's actions sit in a sticky header, the frontmatter "Properties" block is collapsible with an aligned key/value grid, and the inbox drawer and calendar day panel gain an in-app "open" alongside "Open in editor" (#413, #415). The Weekly review's Wins step now shows the week's completed actions as celebration cards, and every log-card surface gains a remembered Oldest/Newest-first toggle (#412). And two vault-graph/card fixes: a bare folder wikilink `[[portfolios/<slug>]]` now resolves to the folder's `_index.md`, so every portfolio link in a log clicks (previously dead), and an overflowing Home project-card next-action caps at a fixed height with a more/less toggle (#414). v0.25.1 fixes the v0.25.0 mouse back/forward feature on macOS: the side buttons did nothing because this Mac's mouse fires them as horizontal **swipe gestures** (`NSEventTypeSwipe`), not `otherMouseUp` buttonNumber 3/4 — root-caused live with an instrumented dev build. The native `NSEvent` monitor now maps a swipe's `deltaX` direction to back/forward (verified against the mouse) while keeping the buttonNumber path, and trackpad page-swipes navigate too (#410). v0.25.0 adds two desktop conveniences. **Back/forward navigation** — `⌘[`/`⌘]` (Ctrl elsewhere) and the mouse's back/forward side buttons step through view history like a browser; since macOS WKWebView doesn't deliver those mouse buttons to the web layer (tauri#10936), a native `NSEvent` monitor bridges them via `nav://back`/`nav://forward` events, with the keyboard path working on every platform (#407). And a **Settings panel** on `⌘,` (the macOS Preferences convention): a calm modal consolidating the app-level preferences that had been blind footer toggles — theme as a segmented control showing the current choice, a labelled metrics switch, and a jump to the vault config editor (#408). v0.24.0 continues the desktop-UI presentation work. The shell **note reader** (the slide-in opened from a wikilink or backlink) now shares the calendar's sectioned rendering via a shared `SectionedBody`, so a note reads identically wherever it opens — titled `##` sections and the `## Logs` history as timestamped cards (the reader omits the calendar's fixed-height Logs cap, since its own panel is the single scroll) (#404). Under the hood, two open-coded idioms were consolidated into shared primitives — the `shortDate` formatter (seven call sites → `lib/dates`) and the uppercase section-label heading (~23 sites → `components/ui/SectionHeading`) — a byte-identical refactor with no behaviour change (#405). v0.23.0 is a desktop-UI presentation pass. A note's frontmatter in the reader moves out of a bare chip row (which read like the note's first heading) into a distinct, labelled `MetaPanel` "Properties" block, and a new reusable `LogCard` gives timestamped log lines a "when + what" hierarchy — replacing the flat bullet list under a project's *Recently in your logs* (#401). The desktop **calendar** now presents a day's note in sections rather than as a raw markdown blob: it parses the note (frontend-side; the domain still hands back one blob) into the separated frontmatter strip, each `##` section under a clear title, and the append-only `## Logs` history as a scrollable stack of timestamped log cards (wikilinks preserved) — and demotes the month grid from a fixed column to a hideable **Pick a date** picker so the note leads. Weekly and monthly notes get the same sectioned treatment (#402). Pure view changes — no domain or wire-format change. v0.22.0 bundles three vault-graph refinements: **note slugs are now globally unique** (#225) — creating a project, action, commitment, question, or custom note whose slug is already taken anywhere in the vault suffixes it `-2`/`-3`/… instead of erroring, so a note that later relocates (an action archived to `_done/`, a project parked) keeps its `[[type/slug]]` backlinks rather than degrading on a stem collision (stewardships excluded — they don't relocate and their flat-vs-expanded guard is worth keeping; External `config.toml` hand-edits now also apply **live**: the app rebuilds its vault on the fly and — on an invalid edit — keeps the last good config and shows a calm, non-red banner. It also fixes a staleness off-by-one (#379): stuck-project "state untouched for N days" is now counted in local days, not UTC, so a project touched today is no longer flagged one day stale in the hours after local midnight. v0.18.0 added **editing `config.toml` from the desktop app** (#365, PRs 1-3): a new **Config** view (Browse group) that shows the raw config, type-checks it against the same validation `Vault::new` runs, and lets you edit and save it in-app. The save path validates FIRST as a hard precondition (an invalid config is never written, so the vault can never be bricked), compare-and-swaps on a content hash to catch a concurrent hand-edit, writes verbatim (comments preserved), and applies the change LIVE — the managed vault is now hot-swappable (`ArcSwap`), rebuilt on the same store/index without a restart. v0.17.0 completes **config-driven frontmatter** (#301 Phase 2): a generic `set_frontmatter` setter — CLI `cdno frontmatter set <note> <key> <value>` and the `set_frontmatter` MCP tool (45th) — writes a typed, `settable`-declared field and its `.cuaderno/index.db` row in one atomic transaction (no hand-edit, no index desync), rejecting unknown/non-settable/reserved (`type`/`status`/period-key) fields and type-checking the value; a `log_on_change` field also stamps a `key: old -> new` line into today's daily. ## What works today

Reachable from the terminal via `cdno`:

- `init` — scaffold a vault
- `log` / `lint` / `capture` — daily-log writes, validation, inbox capture
- `project create / state / park / activate / list / show / milestone {add,done} / waiting {add,resolve}`
- `action add / promote / complete / list` (bullet form + manifest note form)
- `commit create / complete` and `commitments` aggregated view
- `orient` / `status` — morning views
- `portfolio create / list / show` and `file` (file evidence into a portfolio; `--attach <path>` files a non-markdown artefact — PDF/image/video — by copying it in and scaffolding a linked evidence stub, `--move` to relocate instead, #154)
- `question create / park / answer / retire / activate` and `questions` (active grouped by domain)
- `stewardship create / list / show / add-periodic` and `track <activity>`
- `search <query>` — full-text content search across the vault, ranked best-first, with optional `--type` / `--from` / `--to` / `--portfolio` / `--limit` filters
- `completions <shell>` — emit a shell-completion shim (bash, zsh, fish, elvish, powershell) with **dynamic vault-aware tab completion** for slug-valued flags (`--project`, `--portfolio`, `--stewardship`, `--slug` on project/question verbs)

Reachable from Claude via MCP (`cdno-mcp` binary):

- **Context reads (9)** — `get_orientation`, `get_active_questions` (optional domain filter), `get_portfolio_contents`, `get_weekly_context` (ISO-week logs + completed actions + state changes + 2-week commitments), `get_monthly_context` (30-day wins + active questions + portfolios + stuck projects + stewardships + 6-week commitments + project slot allocation), `get_project_context` (project map + 30-day daily-log mentions + body backlinks + resolved core_question), `get_stewardship_tracking` (per-stewardship per-activity tracking notes in a configurable window like `30d`/`6m`/`1y`), `read_daily_note` (a day's markdown, or `exists: false` when none yet), `search_notes` (#172 — FTS5 full-text search over title + body, ranked best-first, with optional note-type / date-window / portfolio filters)
- **Operations** — `append_to_log`, `file_to_portfolio`, `update_project_state`, `add_action` (with optional `with_note`), `promote_action`, `complete_action`, `create_commitment`, `complete_commitment`, `create_tracking_entry` (with optional `routine`), `upsert_daily_section` (write a `{Standup, Intention, Agenda, Meeting}` section — replace, or `append` for live meeting notes)
- **Structural creation (#162)** — `create_project` (active below the cap, parked at/above it), `create_portfolio`, `create_question` (research/life), `create_stewardship` (flat or expanded)
- **Lifecycle (#166)** — `park_project`, `activate_project` (cap-enforced), `set_question_status` (active/parked/answered/retired), `add_periodic_commitment` (recurrence + next date)

Each operation returns a `WriteResultDto { path, message }` so clients can chain on the touched file path.
